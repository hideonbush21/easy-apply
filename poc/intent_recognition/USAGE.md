# 申请状态机测试台 — 使用说明

## 快速启动

```bash
cd poc/intent_recognition

# 1. 配置 API Key
cp .env.example .env
# 编辑 .env，填入你的 KIMI_API_KEY

# 2. 安装依赖
pip install -r requirements.txt

# 3. 启动
python demo.py
```

---

## 主菜单

```
主菜单：
  [1] 模拟手动操作
  [2] 模拟邮件识别（调用 Kimi API）
  [r] 重置所有申请状态
  [s] 查看当前申请状态
  [q] 退出
```

## 置信度三档规则

测试置信度效果的方法：在 `mock_data.py` 中修改邮件内容，使 LLM 返回不同的置信度，或直接在代码中临时 mock。

| 置信度范围 | 触发场景 | 系统行为 |
|-----------|---------|---------|
| ≥ 0.90 | 措辞明确的录取/拒绝/面试邀请 | ✅ 自动执行状态变更 |
| 0.70 – 0.89 | 邮件较模糊或措辞不够直接 | ⚠️ 推「待确认卡片」，用户点击后执行 |
| < 0.70 | 邮件无法判断意图 | 📅 仅建日历 Event，不改状态 |

**候补转正特殊阈值：** 0.85（因学校 waitlist 邮件措辞通常很明确，阈值可适当降低）

## 终态保护测试

**操作步骤：**

1. 先用邮件模拟将 MIT 从「已提交」变更为「已录取」
2. 再次选择 MIT，尝试任何操作

**预期结果：**

```
当前状态：「已录取」
🔒 「已录取」为终态，手动操作受限（仅允许强制人工确认流程）
```

或通过邮件触发：

```
🚫 操作被拦截
   已录取 → 已拒绝
   原因：当前状态「已录取」为终态，任何自动跳转均被拦截，必须强制人工确认
```

## 重置与重复测试

每次测试后状态会保留在内存中（重启程序恢复初始值）。

在不重启的情况下重置：

```
选择 > r
✔ 所有申请状态已重置
```

## 已验证的状态机规则

| 规则 | 验证命令 | 结果 |
|------|---------|------|
| 终态保护 | `已录取` + 任意邮件触发 | blocked |
| 候补转正阈值 | confidence=0.84 vs 0.85 | needs_confirmation / auto_execute |
| 置信度三档 | 0.55 / 0.75 / 0.91 | event_only / needs_confirmation / auto_execute |
| email_only 防手动 | `已提交` 的手动可选 | 空列表 |
| 逆向跳转 | `已提交` → `待申请` | blocked |
